# Memory Flesh Protocol - 记忆固化协议

> 版本: v1.0  
> 创建: 2026-04-01  
> 用途: 规范对话状态保存流程，防止上下文丢失

---

## 一、触发条件

### 1.1 用户显式触发
- "换窗口"
- "上下文极限"
- "保存对话"
- "记忆固化"

### 1.2 系统主动触发
- 对话轮次 > 20 轮
- 单条消息长度 > 4000 字符
- 检测到关键决策点（"就这么定了"/"记下来"等）
- 定时提醒（每1小时检查）

---

## 二、保存优先级

| 优先级 | 内容类型 | 说明 |
|--------|----------|------|
| P0 | 逐条意图 | 每条用户回复的核心意图拆解 |
| P1 | 关键决策 | 确定的方案、选择、结论 |
| P2 | 背景信息 | 项目背景、参考资料、约束条件 |
| P3 | 过程记录 | 完整的对话流、迭代过程 |

---

## 三、文件结构

### 3.1 当日详细摘要
**路径**: `memory/YYYY-MM-DD-full.md`

**内容模板**:
```markdown

## 元信息
- 对话轮次: N
- 关键主题: xxx
- 最终状态: 已完成/待续

## 逐条摘要
### 第N条 - 用户
**内容**: xxx
**意图拆解**: 
1. xxx
2. xxx
**核心意图**: xxx

### 第N条 - 我
**内容**: xxx
**用户反应**: 满意/纠正/追问

## 关键决策
| 决策点 | 结论 | 时间 |
|--------|------|------|
| xxx | xxx | xxx |

## 待办/待续
- [ ] xxx
```

### 3.2 长期记忆更新
**路径**: `MEMORY.md`

**更新时机**:
- 用户偏好变更
- 工作模式调整
- 重要项目里程碑
- 关键教训/经验

**更新格式**:
```markdown
## YYYY-MM-DD 更新

### 新增
- xxx

### 修正
- xxx → xxx

### 删除（过期）
- xxx
```

---

## 四、关键原则

### 4.1 意图优先
- 记录"为什么这么说" > "说了什么"
- 挖掘用户背后的真实需求
- 标注用户的情绪/紧迫度

### 4.2 逐条记录
- 不合并、不省略
- 每条对话独立成段
- 保持时间顺序

### 4.3 版本追踪
- 文件命名: `主题_v1_v2_日期.md`
- 重大变更保留历史版本
- 使用 diff 标注修改点

### 4.4 数据完整
- 不丢任何用户输入
- 保留完整的上下文引用
- 链接到原始文件位置

---

## 五、执行流程

```
触发保存
    ↓
[1] 读取 memory/YYYY-MM-DD-full.md（如有）
    ↓
[2] 追加新对话逐条摘要
    ↓
[3] 提取关键决策 → 更新 MEMORY.md
    ↓
[4] 如有项目文件 → 同步到 03 文件夹
    ↓
[5] 确认保存完成
```

## 六、质量检查清单

- [ ] 是否包含所有用户消息
- [ ] 是否拆解了每条意图
- [ ] 关键决策是否突出显示
- [ ] 文件路径是否正确（file:// 格式）
- [ ] MEMORY.md 是否需要更新
- [ ] 03 文件夹是否需要同步
